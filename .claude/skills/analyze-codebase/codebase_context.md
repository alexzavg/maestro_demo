## Purpose & Stack

This repository is a **mobile UI automation framework demo built on Maestro** (mobile-dev-inc), testing the **Wikipedia app** on Android (primary) and iOS (secondary). It is not an app codebase — the apps under test are prebuilt binaries checked into the repo.

- **Test framework:** Maestro CLI **2.6.1** via Homebrew (`mobile-dev-inc/tap/maestro`); scripts invoke `$HOME/.maestro/bin/maestro`, which is a symlink to `/opt/homebrew/bin/maestro` (the old curl-installed 1.41.0 is kept as `~/.maestro/bin/maestro-1.41.0.bak`)
- **Tooling/runner layer:** Node.js ESM scripts (`package.json` `"type": "module"`), devDependencies: `@devicecloud.dev/dcd`, `dotenv`, `maestro`, `nodemailer`, `open`
- **App under test (Android):** `org.wikipedia` — APK at `apps/android/wikipedia/wikipedia.apk` (Git LFS)
- **App under test (iOS):** `org.wikimedia.wikipedia` — official Maestro-samples simulator build (v7.8.8, universal x86_64+arm64) at `apps/ios/wikipedia/wikipedia.zip` (Git LFS); the unpacked `Wikipedia.app` is generated/gitignored and auto-(re)created by `.scripts/ios/download_app.sh` (falls back to downloading from `https://storage.googleapis.com/mobile.dev/samples/wikipedia.zip`)
- **Optional local LLM tooling:** `ollama:*` npm scripts (serve/run `dolphin-mistral`/kill), plus `maestro:chat` and an AI report (`report:ai:open`)
- **MCP integration:** project-scoped `.mcp.json` registers the built-in Maestro MCP server (`${HOME}/.maestro/bin/maestro mcp --viewer-port 10001`, STDIO). It exposes device/automation tools to Claude Code: `list_devices`, `start_device`, `launch_app`, `stop_app`, `tap_on`, `input_text`, `back`, `take_screenshot`, `inspect_view_hierarchy`, `run_flow`, `run_flow_files`, `check_flow_syntax`, `cheat_sheet`, `query_docs`. Approve the server when prompted at session start. While an MCP session is active, the **Maestro Viewer** (live device screen + flow command sequence) is served at `http://127.0.0.1:10001/` (pinned; Maestro's default is the first free port in 9999–11000). It shows MCP-driven executions only — plain CLI `maestro test` runs don't appear there (use Maestro Studio, port 9999, for interactive inspection). ⚠️ Running two Maestro instances at once (MCP + CLI test/Studio) causes device-driver contention (`Failed to connect to /127.0.0.1:7001` on iOS, "App not installed" on Android).

## Test Architecture

**Flows** are Maestro YAML files named `*.spec.yaml` under `.maestro/flows/<app>/<platform>/<category>/`. Platforms: `android/` and `ios/`; categories: `main/` and `conditional/`. iOS flows mirror the Android ones 1:1. All current flows are tagged `smokeTest`.

**Workspace configs:**
- `.maestro/config.yaml` (Android): `flows/wikipedia/android/{main,conditional}/*.spec.yaml`
- `.maestro/config.ios.yaml` (iOS): `flows/wikipedia/ios/{main,conditional}/*.spec.yaml`
- Both: `excludeTags: util, recordTemplate`, `timeout: 10000`, `testOutputDir: test-results`

**Page Object pattern (key convention):** selectors are not inlined in YAML. Each screen has a JS file in `.maestro/src/pages/<platform>/` that assigns a selector map onto Maestro's `output` object, e.g. `output.launchStepper = { skipBtn: "org.wikipedia:id/fragment_onboarding_skip_button", ... }` ([android/LaunchStepper.page.js](.maestro/src/pages/android/LaunchStepper.page.js)). Flows load them with `runScript: file: "../../../../src/pages/<platform>/<Page>.page.js"` and reference elements as `${output.launchStepper.skipBtn}`. Android selector values are resource-ids or text/regex patterns; iOS selector values are accessibility ids (e.g. `ftux-puzzle-globe`, `tabbar-explore`) or accessibility labels/text (e.g. `"Skip"`, `".*The free encyclopedia.*"`).

**Parameterization:** flows declare `appId: ${appId}`; the value is passed at run time via `-e appId=org.wikipedia` (Android) or `-e appId=org.wikimedia.wikipedia` (iOS) — see the `maestro:test:wikipedia*` scripts and CI workflows.

**Conditional execution:** onboarding handling uses `runFlow: when: visible: id: ${output.launchStepper.skipBtn}` so flows work whether or not the onboarding stepper appears (`MainScreen.spec.yaml`, `LaunchStepper.spec.yaml`).

**Demo failure:** `Third Flow.spec.yaml` (both platforms) asserts a literal `"invalid_id"` — it exists to demonstrate a failing test in reports; do not "fix" it. Consequently `maestro:test:wikipedia:ios` exits non-zero by design (2 passed / 1 failed).

## NPM Scripts Reference

Environment lifecycle (recommended cycle: cleanup → setup → test → teardown), symmetric per platform:

| Script | What it does |
|---|---|
| `env:cleanup:android` | teardown + kill ports/processes/emulators + delete artifacts |
| `env:setup:android` | start Android emulator (`maestro start-device`, API 34, en_US) + install APK |
| `env:teardown:android` | uninstall app + stop emulator |
| `env:cleanup:ios` | teardown:ios + factory-reset simulator + kill ports/processes + delete artifacts |
| `env:setup:ios` | ensure app build (download/unpack) + boot simulator + install app |
| `env:teardown:ios` | uninstall app + stop simulator |

Test execution & authoring:

| Script | What it does |
|---|---|
| `maestro:test:wikipedia:android` | `maestro test .maestro --config .maestro/config.yaml -e appId=org.wikipedia --format html --output test-results/wikipedia.html` |
| `maestro:test:wikipedia:ios` | `maestro test .maestro --config .maestro/config.ios.yaml -e appId=org.wikimedia.wikipedia --format html --output test-results/wikipedia-ios.html` |
| `maestro:inspect` | Maestro Studio (element inspector, port 9999) |
| `maestro:record:wikipedia:android` | Record LaunchStepper flow to `recordings/LaunchStepper.mp4` |
| `record:all:android` | Delete recordings, then `node .scripts/record-flows.js` (records every ANDROID flow) |
| `record:all:ios` | Same recorder against the iOS tree with `RECORD_FAILURES_ONLY=1` — re-runs all iOS flows via `maestro record` and keeps `.mp4`s of FAILED flows only (`{flow}-failed.mp4` in recordings/, picked up by the custom recordings report) |
| `maestro:chat` | `maestro chat` (AI assistant) |
| `test:ios` | One-shot iOS cycle via `.scripts/ios/ios_test_runner.sh`: env:setup:ios → test → env:teardown:ios |

Device/app management — Android: `maestro:device:start:android`, `maestro:device:stop:android` (`adb emu kill`), `android:install:wikipedia` (`adb install -t -r -g`), `android:uninstall:wikipedia`. iOS: `maestro:device:start:ios` / `maestro:device:stop:ios` (simctl boot/shutdown via `.scripts/ios/`), `ios:download:wikipedia`, `ios:install:wikipedia` / `ios:uninstall:wikipedia` (`xcrun simctl (un)install booted`), `ios:erase:simulator`. Target simulator defaults to "iPhone 16 Pro"; override with the `IOS_SIMULATOR_NAME` env var.

Cleanup/kill: `kill:ports` (`.scripts/kill_ports.sh` — ports 5037, 5554–5561, 5584, 5585, 9999, 2077), `kill:processes` (`.scripts/kill_processes.sh` — pkill maestro, qemu-system, emulator, adb, Android Emulator, AndroidStudio, studio), `kill:emulators` (adb emu kill per device).

Reports & artifacts: `report:generate`, `report:serve` (port 2077, recordings-based Android report; deletes test-results first), `report:serve:ios` (port 2077, serves `test-results/wikipedia-ios.html` as index via `REPORT_INDEX`; does NOT delete test-results), `report:open`, `report:open:ios`, `report:ai:open` (opens `test-results/*/ai-report-*.spec.html`), `report:delete`, `report:archive`, `report:email`, `send:report:emails` (archive + email), `recordings:delete`, `artifacts:delete`.

Logs: `capture:android:logs` (adb logcat → `android_emulator_logs.txt`), `capture:android:logs:stop`.

Local LLM: `ollama:serve`, `ollama:run` (dolphin-mistral), `ollama:kill`.

## Reporting Pipeline

1. `record-flows.js` runs flows via Maestro and saves `.mp4` recordings to `recordings/`, named `{flowName}-passed.mp4` / `{flowName}-failed.mp4` (status suffix convention).
2. `generate-report.js` reads `recordings/`, derives pass/fail from filename suffixes, and builds an HTML report at `test-results/test-report.html`.
3. `serve-report.js` regenerates and serves the report on **port 2077** (kills the port first).
4. `archive-report.js` zips `recordings/` + `test-results/` into `test-report/`.
5. `send-report-email.js` emails the newest zip in `test-report/` via Gmail (nodemailer), timestamped `DD-MM-YYYY_HH:MM`.

Required `.env` variables (names only — values are secrets): `EMAIL_USER`, `EMAIL_PASS`, `MANAGER_EMAIL`, `LEAD_EMAIL`. Recipients are manager + lead.

Maestro's own HTML result additionally lands at `test-results/wikipedia.html` (from `maestro:test:wikipedia:android`); AI reports appear as `test-results/*/ai-report-*.spec.html`.

## CI/CD Workflows

All four workflows are **manual (`workflow_dispatch`) only**, Android only, `ubuntu-latest`:

- **android-tests-no-cloud.yaml** — local headless emulator run: enables KVM, Temurin Java 21, installs Maestro CLI, then `reactivecircus/android-emulator-runner@v2` (API 34, x86_64, google_apis, no-window/swiftshader) runs the tests.
- **android-tests-no-cloud-report.yaml** — same emulator setup (with `lfs: true` checkout) but runs the full reporting pipeline: builds `.env` from GitHub secrets, `npm ci`, `record:all:android` → `report:generate` → `archive-report.js` → `send-report-email.js`, uploads `test-results` as an artifact, and publishes recordings/reports to the **gh-pages** branch via `peaceiris/actions-gh-pages@v4`.
- **android-tests-maestro-cloud.yaml** — `mobile-dev-inc/action-maestro-cloud@v2`: uploads the APK + `.maestro` workspace to Maestro Cloud (secrets `MAESTRO_API_KEY`, `MAESTRO_PROJECT_ID`), API 34, en_US, `exclude-tags: util,recordTemplate`, timeout 60000.
- **android-tests-dcd-cloud.yaml** — `devicecloud-dev/device-cloud-for-maestro@v1` (secret `DCD_API_KEY`): pixel-6, API 34, `include-tags: smokeTest`, `exclude-tags: util,recordTemplate`, retry 1, HTML report, downloads ALL artifacts.

## iOS Testing

Implemented 2026-07-14 as a full mirror of the Android lifecycle (no hardcoded machine paths; verified end-to-end).

- **App under test:** official Maestro-samples Wikipedia iOS simulator build v7.8.8 (`org.wikimedia.wikipedia`, universal x86_64+arm64, min iOS 16.6). Source of truth is `apps/ios/wikipedia/wikipedia.zip` in Git LFS; `.scripts/ios/download_app.sh` unpacks it (and re-downloads from `https://storage.googleapis.com/mobile.dev/samples/wikipedia.zip` if the zip is missing or is an un-pulled LFS pointer). The unpacked `Wikipedia.app` is gitignored.
- **Simulator:** default **iPhone 16 Pro** (first available match by name; override via `IOS_SIMULATOR_NAME`). Boot waits on `xcrun simctl bootstatus -b`.
- **Run cycle:** `env:cleanup:ios` → `env:setup:ios` → `maestro:test:wikipedia:ios` → `report:serve:ios` (or `report:open:ios`) → `env:teardown:ios`/`env:cleanup:ios`. Cleanup factory-resets the simulator (`simctl erase`) so each run starts on a brand-new device. `npm run test:ios` does setup → test → teardown in one shot (non-interactive, propagates the test exit code).
- **Flows/selectors:** `.maestro/flows/wikipedia/ios/` + `.maestro/src/pages/ios/`, driven by `.maestro/config.ios.yaml`. Selectors verified against the live app: onboarding uses `Skip`/`Next`/`ftux-puzzle-globe`/`"page 1 of 4"`; Explore screen uses `Search Wikipedia`, `tabbar-explore`, `profile-button`. MainScreen flow conditionally dismisses the "Year in Review" popup and the "logged out" auth modal (patterns from Maestro's own sample flows).
- ⚠️ **Driver contention gotcha:** running the iOS suite while another Maestro instance (Studio, MCP session) holds the simulator causes `Unable to clear state ... Failed to connect to /127.0.0.1:7001` failures. Close other Maestro instances first (same class of issue as the Android "App not installed" gotcha).

## Conventions & Gotchas

- **Run cycle:** always `env:cleanup:<platform>` → `env:setup:<platform>` → `maestro:test:wikipedia:<platform>` → `env:teardown:<platform>` (`:android` or `:ios`). Cleanup before each run avoids stale ports/processes/emulators (README "IMPORTANT" notes).
- **Tags:** `smokeTest` on all real flows; `util` and `recordTemplate` are reserved exclusion tags (excluded in config and every cloud CI run).
- **package.json script ordering convention:** Android-related scripts first, then iOS-related scripts, then shared utilities (inspect/chat, generic reports, delete/kill, ollama). Keep new scripts in their platform group.
- **Known issue:** "App not installed" error when Maestro Studio is open — Studio holds the same emulator; close Studio and re-run (README Troubleshooting, upstream issue mobile-dev-inc/Maestro#1104).
- **`Third Flow.spec.yaml` fails on purpose** (asserts `invalid_id`) — a demo of failure reporting.
- **Selectors live only in page objects** (`.maestro/src/pages/<platform>/*.page.js`); when adding a flow, create/extend a page file and load it with `runScript` rather than hardcoding ids in YAML.
- **Generated dirs** (`test-results/`, `recordings/`, `test-report/`, `android_emulator_logs.txt`) are disposable artifacts — safe to delete via the `*:delete` scripts.
- `.windsurf/rules/` contains AI-assistant policies for this repo: scan the codebase before answering, consult official Maestro docs for framework questions, persist codebase memory, and **never modify files unless explicitly asked**. These reflect the owner's expectations for AI tooling here.
- Scripts assume **macOS/zsh + Homebrew Maestro install** locally (Windows-specific commands were deliberately removed; see git history).
