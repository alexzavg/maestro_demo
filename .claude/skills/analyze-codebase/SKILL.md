# Analyze Codebase

You are analyzing a mobile UI automation framework built on **Maestro** (mobile-dev-inc). Your goal is to produce — or refresh — a single, durable memory file that captures the *main concepts of the setup* so that any future skill or session can load it and immediately understand how this repository works without re-scanning it.

## Output

Write the result to `.claude/skills/analyze-codebase/codebase_context.md` (overwrite if it exists). This file is the shared, general-purpose memory for this codebase and will be consumed by future skills — write it for an AI/engineer audience: factual, specific, path-referenced, no filler.

## How to analyze

Scan the repository **excluding** binary/app payloads (`apps/**/Wikipedia.app`, `*.apk`, `node_modules`, `.git`, media files) — read their locations, not their contents. Cover, at minimum:

1. **Purpose & stack** — what the repo is, the framework(s) in use (Maestro CLI, Node.js tooling), target platforms (Android/iOS), and the app(s) under test (name, package/bundle id, where binaries live).
2. **Test architecture** — where flows live (`.maestro/flows/**`), how they are organized (folders, tags, naming like `*.spec.yaml`), the Maestro config (`.maestro/config.yaml`: included flows, excluded tags, timeouts, output dir), and the Page Object pattern used via `runScript` + JS page files in `.maestro/src/pages/` (`output.<page> = {...}` selector maps, referenced as `${output.page.element}`). Note parameterization (e.g. `-e appId=...`) and conditional flows (`runFlow.when.visible`).
3. **Runner & lifecycle tooling** — every meaningful `package.json` script grouped by concern: environment setup/cleanup/teardown, device/emulator management, app install/uninstall, test execution, recording, inspection (Maestro Studio), log capture, and process/port kill scripts in `.scripts/` (list which ports/processes they target).
4. **Reporting pipeline** — how HTML reports are generated, served, archived, and emailed (`.scripts/generate-report.js`, `serve-report.js`, `archive-report.js`, `send-report-email.js`), the directories involved (`test-results/`, `recordings/`, `test-report/`), naming conventions (e.g. `-passed`/`-failed` suffixes on recordings), and required env vars from `.env` (names only — NEVER copy secret values).
5. **CI/CD** — each GitHub Actions workflow in `.github/workflows/`: trigger, what it runs (local emulator vs Maestro Cloud vs Device Cloud), and notable setup steps (KVM, Java version, emulator runner action, API level).
6. **iOS path** — how iOS tests run (`.scripts/ios/ios_test_runner.sh`): simulator used, where the `.app` comes from, and any machine-specific hardcoded paths worth flagging.
7. **Conventions & gotchas** — recommended run cycle (cleanup → setup → test → teardown), known issues from README troubleshooting, tags in use (`smokeTest`, `util`, `recordTemplate`), intentionally failing/demo flows, and any other rules found in repo config (e.g. `.windsurf/rules/`).

## Memory file format

Structure `codebase_context.md` as:

```markdown
# Codebase Context — <repo name>
> Generated by the analyze-codebase skill on <date>. Refresh by re-running the skill.

## Purpose & Stack
## Repository Map        (annotated directory tree of the meaningful paths)
## Test Architecture     (flows, config, page-object pattern, selectors, tags)
## NPM Scripts Reference (grouped table: script → what it does)
## Reporting Pipeline
## CI/CD Workflows
## iOS Testing
## Conventions & Gotchas
```

## Rules

- Be exhaustive on concepts, terse on prose — every claim must be backed by a real file path you inspected.
- Names of environment variables are fine; secret **values** must never appear in the memory file.
- If something has changed since the last generation, rewrite the file wholesale rather than patching it.
- Finish by confirming the file path written and a one-paragraph summary of what the codebase is.
